# 交互走查

## 概述

对可访问的前端页面进行交互走查，核对 PRD、交互设计文档、设计稿和实际实现之间的差异，并输出可追踪、可复查、可分版本管理的审查报告。

本技能是平台无关的：优先使用当前平台内置浏览器能力；没有内置能力时，选择可用的浏览器自动化、MCP 或截图工具；没有任何网页访问能力时，只能做文档走查，不能声称已完成页面交互验证。

## 输入要求

至少需要一个可执行审查的对象：

- 网页链接、本地预览地址、可启动项目路径，三者至少其一
- 推荐同时提供 PRD、交互设计文档、设计稿、原型、截图、账号、测试范围、目标设备和浏览器

输入材料不完整时允许降级，但必须在报告中标注“无法验证项”和“降级原因”。

## 先澄清再执行

以下情况必须先澄清，不能直接开始：

- 没有网页链接、本地页面、可启动项目路径，也没有截图或设计稿
- 没有说明审查全部页面还是指定页面，且无法从输入材料推断
- 需要登录但缺少账号、权限或测试环境说明
- 当前平台没有可用浏览器、MCP、截图或网页访问工具
- 设计稿、PRD、交互文档之间明显不是同一版本
- 用户要求“对照设计稿/PRD”，但没有提供对应材料或可访问链接

澄清时一次优先问最阻塞的问题；如果平台支持多轮表单或提问工具，可用它收集范围、材料、账号、设备和输出路径。

## 工具选择

按顺序选择可用工具，不要写死某个平台：

1. 先探测当前平台公开可用的网页访问能力，再选最合适的浏览器路径，例如 Codex Browser 插件、可视化浏览器、截图能力
2. 已安装或已暴露的浏览器 skill / MCP，例如 agent-browser、browser-use、Playwright MCP、Puppeteer MCP、Selenium MCP
3. 本地自动化库或命令行工具，例如 Playwright、Puppeteer、Selenium、Chrome DevTools Protocol
4. 只能访问文档或静态截图时，做文档审查或静态走查，并明确未进行真实交互验证

在 Codex 中，如果用户要求打开、点击、截图或验证本地网页，优先使用 Codex 自带 Browser 插件；如果需要脚本化批量走查，再考虑 Playwright 或 agent-browser。

## 工作流

### 1. 建立审查范围

解析输入材料并列出：

- 页面清单和 URL
- 用户路径和关键任务
- 需要对照的 PRD 条目、交互规范、设计稿节点
- 目标设备、浏览器、断点、登录角色
- 本轮审查版本号和报告输出位置

如果用户没有指定版本号，使用 `v1`；同一对象重复审查时递增为 `v2`、`v3`。
如果 PRD、交互设计文档、设计稿之间存在版本冲突，先记录冲突并请用户确认最新基准版本；在未确认前不要擅自把某一份材料当成唯一真相。

### 2. 拆解审查矩阵

对每个页面整理检查点：

- 页面进入、加载、刷新、返回、深链访问
- 表单输入、校验、提交、取消、重置
- 按钮、链接、菜单、标签页、筛选、排序、分页、搜索
- 弹窗、抽屉、Toast、气泡、确认框、二次确认
- 默认态、悬停态、聚焦态、点击态、选中态、禁用态、加载态、空态、错误态、成功态
- 权限差异、异常输入、网络失败、接口异常、会话过期
- 响应式布局、键盘可达性、焦点顺序、可读性
- 文案、字段、数据展示、埋点或业务规则

详细检查表见 `references/checklist.md`。只有需要展开逐项核查时再读取。

### 3. 执行页面走查

逐个页面访问并记录证据：

- 打开页面并确认可访问状态
- 截取关键状态截图，保存路径或工具引用
- 执行关键用户路径，不只停留在首屏观察
- 对照 PRD、交互文档、设计稿记录差异
- 对每个问题写清楚期望、实际、复现步骤、影响和依据
- 对无法访问、权限不足、环境异常的页面单独标注，不把它们算作通过

若设计稿来自 Figma、飞书、Sketch、图片或 PDF，优先使用平台可用的读取、截图或导出能力获取可比较证据。无法读取时，请用户补充截图或可访问权限。

### 4. 分级问题

使用以下优先级：

- `P0 阻塞`：核心流程不可用、无法提交、无法进入页面、数据严重错误、权限越权
- `P1 高`：主要任务受影响、关键状态缺失、与 PRD/交互设计明显冲突
- `P2 中`：体验不一致、反馈不清晰、边界状态缺失、响应式明显问题
- `P3 低`：文案、间距、轻微视觉还原、非关键优化建议
- `Info`：观察记录、风险提示、无法验证项

不要把所有问题都写成“建议优化”。能复现的问题要写成缺陷；缺少输入或证据的内容写成风险或无法验证项。

### 5. 输出版本化报告

默认输出到：

```text
interaction-walkthrough-reports/<YYYY-MM-DD>-<target>-vN/
```

其中 `<target>` 应做成便于文件系统使用的短标识；如果平台不支持落盘，则保留相同版本号语义，在对话中输出 `vN` 和修改记录。

建议包含：

```text
report.md
change-log.md
evidence.json
screenshots/
```

如果平台不能写文件，则在对话中输出完整报告，并说明未落盘。

报告模板见 `references/report-template.md`。每次复审都要在 `change-log.md` 或报告“版本记录”中记录：

- 本次审查版本
- 输入材料版本或链接
- 已修复问题
- 新增问题
- 遗留问题
- 无法验证项变化

## 输出要求

最终报告必须包含：

- 审查对象、范围、输入材料、工具、时间、版本
- 总体结论和风险摘要
- 按优先级排序的问题清单
- 每个问题的页面、复现步骤、期望、实际、依据、证据和建议
- 页面逐项走查结果
- 状态覆盖情况
- 与 PRD、交互文档、设计稿的差异
- 无法验证项和原因
- 本轮修改记录或与上一版的差异

## 常见错误

- 只看页面截图，没有实际点击主要交互
- 只对照设计稿，忽略 PRD 的业务规则和异常流程
- 没有记录证据，导致问题无法复现
- 没有说明审查范围，用户误以为全站都审过
- 没有区分缺陷、建议、风险、无法验证项
- 没有版本号，复审时无法追踪问题是否修复
- 没有浏览器工具却声称完成了真实交互走查
